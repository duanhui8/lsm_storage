一、SQL 层与存储层的入口接口 —— DefaultHandler
--------------------------------------------------------------------------------
【问题】default_handler.h 是 SQL 层与 oblsm 存储引擎的入口接口吗？

【结论】是。DefaultHandler 是 SQL 层与存储层之间的门面(Facade)接口,全局单例。

  架构位置:
    SQL 层 (execute_stage, session, init)
           │
    DefaultHandler  ← 门面，全局单例 the_default_handler()
           │
    Db  ← 管理 TableEngine (heap 或 lsm)
           │
    Table / TableEngine

  调用方：
    - src/observer/common/init.cpp          系统初始化
    - src/observer/sql/executor/execute_stage.cpp  SQL执行阶段
    - src/observer/session/session.cpp       会话管理

【关键发现】DDL 和 DML 走不同路径:
  - DDL (CREATE TABLE 等): CommandExecutor → DefaultHandler::create_table() → Db → Table
  - DML (SELECT/INSERT 等): PhysicalOperator (Volcano模型) 直接操作 Table,不经过 DefaultHandler

  DefaultHandler 只提供 DDL 接口 (create_table, find_table 等),没有 insert/select/update/delete。
  原因是 Volcano 执行模型使用迭代器模式 (open→next→close),算子直接持有 Table* 调用其方法。


二、OceanBase 中库与表的关系
--------------------------------------------------------------------------------
【问题1】oceanbase 中库和表是什么关系？宏块中如何维护库表关系？

  层次结构:
    Tenant(租户) → Database(库) → Table(表) → Tablet/Partition(分区)
                                          → SSTable(存储文件)
                                                → Macro Block(宏块, 2MB)
                                                      → Micro Block(微块, 16KB)

  宏块头部记录: table_id, tablet_id, data_version, macro_block_id, row_count, rowkey_range
  宏块只记录 table_id,不直接记录 database 信息。库与表的从属关系通过元数据表维护:
    table_id → __all_table → database_id → __all_database → database_name

【问题2】元数据表（schema 层）保存在哪里？

  三层自举机制:
    第一层: 编译期硬编码 — 核心系统表 schema 编译进 observer 二进制,不需要从磁盘读
    第二层: 系统 Tablet — 固定 tablet_id,通过 Superblock 定位
    第三层: Superblock/Bootstrap — 记录系统 Tablet 所在日志流位置

  启动流程:
    Superblock → 系统 Tablet → 用硬编码 schema 解析 → 读出 __all_table 内容 → 访问用户表

【问题3】miniob 的库名是写死 sys 的吗？

  更早版本: 只有 sys 一个数据库,写死。
  当前版本: sys 仍然硬编码为系统库(init 时强制创建+打开),但已支持多数据库:
    - create_db() ✅   drop_db() ✅   list_dbs() ✅
    - sys 受保护不可删除
  这是 MiniOB 的过渡状态,对应 OceanBase 的系统表空间概念但不持久化元数据。


三、DiskBufferPool —— 页面缓存(Buffer Pool)
--------------------------------------------------------------------------------
【问题1】disk_buffer_pool.cpp 是干嘛的？对应 OceanBase 4.4.2 哪个组件？

  功能: 管理数据库页面在内存和磁盘之间的映射,实现页面缓存。
  核心职责: 页面缓存、LRU 淘汰、页面分配/释放(bitmap)、DoubleWrite(防部分写)、崩溃恢复(redo)

  三层架构:
    BufferPoolManager   ← 管理所有 DiskBufferPool, 分配 buffer_pool_id
    DiskBufferPool      ← 单个文件的缓冲池, get_this_page/allocate_page/dispose_page
    BPFrameManager      ← Frame 对象池, LRU 淘汰(LinkedHashMap)

  对应 OceanBase 4.4.2 的组件(非一对一):
    DiskBufferPool        → ObDiskBufferPool / ObBufferPool
    BPFrameManager        → ObBlockCache (块级缓存)
    DoubleWriteBuffer     → ObDoubleWriteManager
    BufferPoolManager     → ObTenantDiskBufferPool

  核心概念对照: Page↔Macro Block, Frame↔Cache Handle, pin/unpin 概念相同

【问题2】disk_buffer_pool 相当于 OceanBase 的 KV Cache 吗？

  不是。两个不同层次的缓存:
    OceanBase: KV Cache (行级缓存,纯读) → Block Buffer Pool (页面缓存) → 磁盘
    MiniOB:   (没有 KV Cache) → DiskBufferPool → 磁盘

  disk_buffer_pool = InnoDB Buffer Pool = OceanBase Block Buffer Pool (同一层)
  KV Cache 是坐在 Buffer Pool 上面的另一层缓存

【问题3】MiniOB 除了 disk_buffer_pool 还有哪些缓存？

  ✅ DiskBufferPool     — 页面缓存,唯一完整实现
  ❌ ObLRUCache         — LSM Block Cache,空壳 stub
  ❌ PlanCacheStage     — 执行计划缓存,空壳 stub (类体为空)
  ❌ QueryCacheStage    — 查询结果缓存,空壳 stub
  ✅ common::LruCache   — 底层 LRU 工具类(被 Buffer Pool 使用)

五、ObKeyBtree —— MemTable 内部的并发 B-Tree
--------------------------------------------------------------------------------
【问题】mystorage 用 std::map 代替了 ObKeyBtree，ObKeyBtree 是什么？

【答案】ObKeyBtree 是 MemTable 内部 ObQueryEngine 用于支持范围扫描的 B-Tree。

  在 MemTable 中的位置:
    ObQueryEngine
      ├── Hash Table (ObMtHash)  ← 点查 O(1)
      └── ObKeyBtree             ← 范围扫描 O(log N)

  关键设计（源码注释 ob_keybtree.h:243-247）:
    "OceanBase's keybtree is a btree(we are not b+tree) that is specially
     optimized for concurrent insertion. It uses the techniques of epoch based
     reclaimation and Copy-on-Write to allow the high capibility under high
     concurrency."

  与普通 B+Tree 的区别:
    类型:      B-Tree (数据可存内部节点) vs B+Tree (数据只存叶子)
    并发写:    Copy-on-Write 无锁 vs 通常用 latch
    内存回收:  Epoch-based (等所有读者离开) vs 写完即释放
    目标:      MemTable 专用内存索引 vs 通用磁盘索引
    节点分配:  CPU 分区分配器 vs malloc/free

  为什么需要特制 B-Tree：
    MemTable 多线程并发写入同一有序结构。
    如果用 std::map + 大锁，写性能极差。
    ObKeyBtree 用 COW + epoch 实现无锁并发读 + 高效写:
      写线程: COW 复制节点路径 → 原子切换指针 → 旧节点不立即释放
      读线程: 持有 hazard pointer 安全遍历旧树
      epoch 结束: 确认无读者后回收旧节点

  规模: 2760 行 (ob_keybtree.h 323行 + ob_keybtree.cpp 1789行 + ob_keybtree_deps.h 648行)

  mystorage 用 std::map<uint64_t, ObMvccRow*> 替代是正确的教学简化:
    目的: 理解 "MemTable 需要有序结构支持 range scan"
    不需要: 让学生陷入 2760 行并发 B-Tree 实现细节


六、关键架构洞察
--------------------------------------------------------------------------------
1. 两个存储引擎并存
   - oblsm: 独立的 LSM-Tree 引擎 (LevelDB 风格,不依赖 observer)
   - mystorage: 对照 OceanBase 4.4.2 精简移植的引擎
   - 老 Heap 引擎: disk_buffer_pool + record_manager (教学基线)
   三者各自独立,通过 TableEngine 策略模式接入 SQL 层

2. mystorage 的设计哲学
   每个文件头部都标了 "Refer to: /opt/oceanbase4.4.2/src/storage/xxx"
   目标是让读者看完 mystorage 的简化版后,能看懂 OB 4.4.2 的生产代码。
   它不是可运行的引擎,而是一份 "OB 4.4.2 存储引擎导读地图"。

3. 缓存分层 (从底层到上层)
   磁盘文件 → Buffer Pool (页面级) → Block Cache (块级, stub) → KV Cache (行级,OB独有)
   MiniOB 只有最底层是完整的,上层都是占位符。
